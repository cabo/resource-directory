Preface

The present test descriptions are meant to guide coordinated interoperability tests of the CoRE Resource Directory draft. When preparing for an interop event, implementors should not try to follow the test description to the letter, but run the implementations they developed based on the draft as they are, and document deviations from the described baseline test data when non-interoperability is found. In particular, the paths of lookup and registration resources on the server can be arbitrary, and the concrete paths, rd/xxx, rd/yyy, /rd/xyz, and /rd-group/zzz, given in the test specifications are to be seen as variable. Clients must always use the paths discovered in earlier tests.

Interoperability Test Description
Identifier: TD_CoRE_resource-directory_01
Objective: Discover RD server with coap multicast
Configuration: CoAP_CFG_resource-directory
References: [resource-directory] 5.2
Pre-test conditions:
  • Server enabled its interface for MC address "All CoAP Nodes" address FF0X::FD
  • Server contains resource /rd
Test Sequence: Step Type Description
1 Stimulus Client is requested to send a MC GET request with:
  • Type = 1 (NON)
  • Code = 1 (GET)
2 Check The request sent by the client contains:
  • Type = 1 and Code = 1
  • Token = XX, set by client
  • Uri-Path: ".well-known", "core"
  • Uri-Query: "rt=core.rd"
  • Accept: 40
  • empty payload
3 Check Server sends response containing:
  • Code = 2.05 (Content)
  • Token = XX
  • Content-Format: 40
  • Payload with RD resource
4 Verify Client displays </rd>;rt="core.rd";ct="40"

Identifier: TD_CoRE_resource-directory_03
Objective: Simple registration
Configuration: CoAP_CFG_resource-directory
References: [resource-directory] 5.3.1
Pre-test conditions:
  • Client knows address of resource directory (server)
  • Server contains resource /rd
  • Client prepares /.well-known/core resource with links
    • </sensors/temp>;ct=41;rt="temperature-c";if="sensor";anchor="coap://spurious.example.com:5683"
    • </sensors/light>;ct=41;rt-"light-lux";if="sensor"
Test Sequence: Step Type Description
1 Stimulus Client is requested to send a POST request with:
  • Type = 0 (CON)
  • Code = 2 (POST)
2 Check The request sent by the client contains:
  • Type = 1 and Code = 2
  • Token = XX, set by client
  • Uri-Path: ".well-known", "core"
  • Uri-Query: "lt=6000", "ep=node1"
  • empty payload
3 Check Server sends response containing:
  • Code = 2.04 (Changed)
  • Token = XX
  • empty payload
4 Continuation Server sends request containing:
  • Type = 1 and Code = 2
  • Token = XY, set by server
  • Uri-Path: ".well-known", "core"
  • Accept: 40
  • empty payload
5 Continuation response client sends response containing:
  • Code = 2.05 (Content)
  • Token = XY
  • Content-Format: 40
  • payload
    • </sensors/temp>;ct=41;rt="temperature-c";if="sensor";anchor="coap://spurious.example.com:5683"
    • </sensors/light>;ct=41;rt-"light-lux";if="sensor"
6 Verify
  • See test TD_CoRE_resource-directory_15
  • Assumption: /rd/xxx is the registration resource
